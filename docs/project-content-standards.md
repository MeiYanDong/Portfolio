# 项目介绍内容标准

这个标准用于维护个人网站的项目数据，避免项目页只像作品清单，而不是能力证据。

## 写作结构

每个项目必须回答五个问题：

1. 这个项目解决什么真实问题。
2. 我在项目里承担什么角色。
3. 我做了哪些关键工程或产品决策。
4. 它现在交付了什么结果或可复用能力。
5. 简历里应该如何用一句话表达。

## 字段规则

- `description`：一句话说清项目对象、使用场景和系统边界。
- `resumeLine`：用动词开头，突出构建、设计、打通、沉淀、部署等贡献。
- `impact`：写结果和价值，不虚构用户数、收入、性能数字。
- `role`：透明说明个人贡献，避免伪装成大型团队项目。
- `highlights`：三条以内，优先写系统能力、工程取舍和工作流闭环。
- `story.problem`：写为什么需要这个项目。
- `story.process`：写我是怎么拆解和实现的。
- `story.result`：写最终形成了什么能力、产品或证据。

## 图片规则

- 首选真实产品截图，尤其是能展示界面、数据结构、工作流的截图。
- 如果没有截图，使用项目专属视觉面板，展示项目名、主题和能力标签。
- 禁止用其他项目的截图替代当前项目。
- 禁止为了视觉统一而复用无关图片。
- 截图不需要覆盖所有细节，但第一眼必须能看出属于这个项目。

## 展示原则

- 重点项目写成小型 case study，早期作品也要补齐问题、过程、结果。
- 没有证据的数字不要写，用“形成工作台”“沉淀流程”“打通链路”等可验证表达。
- 项目介绍优先服务简历和面试追问：别人点进去后，应该能问出技术边界、决策和贡献。
